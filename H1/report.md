## Шаг 1. Анализ требований проекта

### Описание задачи

В качестве практической IT-задачи выбрано создание **Calculator API на FastAPI** — REST API сервиса для выполнения арифметических операций (сложение, вычитание, умножение, деление), развёрнутого в Docker-контейнере с элементами production-grade инженерии.

### Ключевые требования к проекту

| № | Требование | Описание |
|---|-----------|----------|
| 1 | REST API | HTTP-эндпоинт для арифметических операций с JSON-запросами/ответами |
| 2 | Docker-контейнеризация | Multi-stage Dockerfile, docker-compose для production и development |
| 3 | Конфигурация через окружение | Управление параметрами через переменные окружения и `.env`-файлы |
| 4 | Автоматическое тестирование | Unit-тесты и integration-тесты с pytest |
| 5 | Production-grade структура | Слоистая архитектура: routes, services, schemas, config |
| 6 | Логирование | Структурированные логи с настраиваемым уровнем |
| 7 | Документация | README.md с инструкциями, автогенерация OpenAPI/Swagger |

### Требования к безопасности

- Код проекта должен оставаться **локальным** — загрузка в облачные сервисы нежелательна
- Использование непривилегированного пользователя в Docker-контейнере
- Валидация входных данных на уровне API

### Бюджет

- Предпочтение отдаётся **бесплатным и open-source** инструментам
- Допускается использование бесплатных тарифов коммерческих продуктов

### Сложность проекта

Проект средней сложности, включающий:
- Бэкенд-разработку на Python/FastAPI
- Контейнеризацию с Docker
- Настройку CI-подобного окружения (тесты, линтинг)
- Работу с конфигурацией и переменными окружения
- Необходимость доступа к локальной файловой системе для работы с проектом

## Шаг 3. Выбор инструмента

### Выбранный инструмент: Kiro

На основе проведённого сравнительного анализа выбран **Kiro** — IDE-агент на базе VS Code с уникальным spec-driven подходом к разработке.

### Ключевые факторы выбора

1. **Безопасность данных (5/5).** Код проекта остаётся локальным — не загружается в облачные среды. Это критически важно для выполнения требования задания о конфиденциальности кода.

2. **Полный доступ к контексту проекта (5/5).** Kiro работает с локальной файловой системой и имеет полное представление о структуре проекта. Spec-driven workflow формализует требования, дизайн и задачи, что даёт AI-агенту максимально полный контекст для генерации кода.

3. **Spec-driven development.** Уникальная методология Kiro:
   - `requirements.md` — формализация требований с критериями приёмки
   - `design.md` — проектный документ с архитектурой, компонентами, моделями данных
   - `tasks.md` — декомпозиция на задачи с трассируемостью к требованиям
   - Автоматическое выполнение задач по плану

4. **Качество генерируемого кода (5/5).** Благодаря формализованным требованиям и проектному документу, Kiro генерирует код с учётом архитектурных решений, обработки ошибок, тестирования и production-практик.

5. **Система hooks и steering files.** Автоматизация рутинных операций: запуск тестов, линтинг, форматирование при сохранении файлов.

6. **MCP-поддержка.** Возможность расширения функциональности через Model Context Protocol.

### Причины отклонения Cloud-агентов

Cloud-агенты (Bolt.new, Lovable, Replit Agent) были отклонены по следующим причинам:

| Причина | Описание |
|---------|----------|
| **Безопасность данных** | Весь код загружается и обрабатывается на серверах сервиса. Нет контроля над хранением и использованием данных. Для проекта с требованием локальности кода — неприемлемо |
| **Ограниченный контекст** | Cloud-агенты не имеют доступа к локальным файлам проекта. Работа с существующей кодовой базой затруднена |
| **Стоимость** | Бесплатные тарифы сильно ограничены. Для полноценной работы требуется платная подписка |
| **Отсутствие интеграции** | Результат работы нужно вручную переносить в локальный проект. Нет интеграции с Git, терминалом, отладчиком |

### Альтернативные IDE-агенты и причины предпочтения Kiro

#### Cursor (24/30)

Cursor — сильный конкурент с отличной скоростью генерации кода (5/5). Однако Kiro предпочтительнее по нескольким причинам:

- **Отдельная IDE.** Cursor — это форк VS Code, отдельное приложение. Требует миграции с привычного редактора и не является плагином к существующей IDE.
- **Отсутствие spec-driven workflow.** Cursor не предоставляет структурированного подхода к формализации требований и декомпозиции задач. Качество результата сильно зависит от качества промптов пользователя.
- **Стоимость.** Pro-план $20/мес для полноценного использования.

#### GitHub Copilot (23/30)

GitHub Copilot — зрелый продукт с хорошей интеграцией в существующие IDE. Причины предпочтения Kiro:

- **Преимущественно автокомплит.** Copilot ориентирован на построчную генерацию кода, а не на агентный подход к решению задач. Copilot Chat улучшает ситуацию, но не достигает уровня агентности Kiro.
- **Отсутствие spec-driven workflow.** Нет встроенной системы формализации требований, проектирования и декомпозиции задач.
- **Ограниченный контекст.** Контекстное окно ограничено открытыми файлами, не всегда учитывает полную структуру проекта.

#### Cline (25/30)

Cline — мощный open-source агент для VS Code с высокой оценкой по безопасности и стоимости. Причины предпочтения Kiro:

- **Отсутствие структурированного workflow.** Cline — агентный инструмент общего назначения без встроенной системы спецификаций. Нет формализации требований, проектирования и декомпозиции задач.
- **Зависимость от качества промптов.** Без spec-driven подхода результат сильно зависит от того, насколько точно пользователь формулирует задачи.
- **Отсутствие hooks и steering.** Нет встроенной автоматизации (запуск тестов при сохранении, линтинг) и настройки поведения агента через steering files.
- **Необходимость ручного управления API-ключами.** Пользователь самостоятельно настраивает и оплачивает доступ к LLM-провайдерам.

### Итог

Kiro выбран как оптимальный инструмент для данной задачи благодаря сочетанию максимальной безопасности данных, полного доступа к контексту проекта и уникального spec-driven подхода, который обеспечивает структурированную и воспроизводимую разработку.

### Настройка MCP (Model Context Protocol)

Kiro поддерживает MCP для расширения возможностей AI-агента через подключение внешних серверов. Конфигурация MCP выполняется через файл `.kiro/settings/mcp.json`. Для данного проекта дополнительные MCP-серверы не потребовались — встроенных возможностей Kiro (доступ к файловой системе, терминал, веб-поиск) было достаточно для полного цикла разработки.

При необходимости MCP позволяет подключить:
- Серверы документации (например, `aws-documentation-mcp-server`)
- Инструменты для работы с базами данных
- Внешние API и сервисы

### Создание проекта через Kiro Spec Workflow

Проект создан с использованием spec-driven workflow Kiro:

1. **Инициализация спецификации.** Через интерфейс Kiro создана новая спецификация `calculator-api-homework`.
2. **Формализация требований.** Kiro сгенерировал `requirements.md` с 9 группами требований и критериями приёмки в формате WHEN/THEN/SHALL.
3. **Проектирование.** На основе требований Kiro сгенерировал `design.md` с архитектурой, компонентами, моделями данных и свойствами корректности.
4. **Декомпозиция задач.** Kiro сгенерировал `tasks.md` с 11 задачами, каждая со ссылками на требования для трассируемости.

### Использование скила fastapi-templates

Для обеспечения production-grade структуры проекта был создан и применён скил **fastapi-templates** (`H1/calculator-api/.agents/skills/fastapi-templates/SKILL.md`). Скил определяет набор паттернов для создания FastAPI-приложений:

- **Структура проекта:** `app/api/v1/endpoints/`, `app/core/`, `app/services/`, `app/schemas/`
- **Lifespan context manager** вместо устаревших startup/shutdown events
- **Dependency Injection** через `Depends()` для сервисов и конфигурации
- **Service Layer** — бизнес-логика вынесена в класс `CalculatorService`
- **Versioned API Router** — централизованный роутер `api/v1/router.py` с префиксом из конфигурации
- **Паттерн конфигурации** — `core/config.py` с `@lru_cache()` синглтоном

Скил обеспечил единообразие архитектуры и соответствие best practices FastAPI-разработки.

## Шаг 5. Практическое применение

### Описание процесса разработки

С использованием Kiro был создан полноценный Calculator API на FastAPI — от формализации требований до работающего Docker-контейнера.

### Примеры промптов

В процессе работы с Kiro использовались следующие промпты (на русском языке):

**Промпт 1 — Инициализация проекта:**
> «привет есть домашняя работу которую нужно сделать... в качестве IT задачи предлагаю сделать какой то простое API... калькулятор на FastAPI»

Kiro проанализировал задание и предложил структуру спецификации с требованиями.

**Промпт 2 — Уточнение требований Docker:**
> «Требование 5: Docker-контейнеризация вот тут добавь что необходимо чз валум монтировать код чтоб изменения были видны в запущенном контейнере»

Kiro добавил требования для `docker-compose.dev.yml` с volume mount и `--reload`.

**Промпт 3 — Последовательное выполнение задач:**
> «поехали дальше» / «дальше» / «давай делать»

После утверждения спецификации, задачи выполнялись последовательно — каждая команда запускала выполнение следующей задачи из плана.

### Количество итераций

| Этап | Итерации | Описание |
|------|----------|----------|
| Формализация требований | 3–4 | Уточнение требований к Docker, логированию, структуре отчёта |
| Утверждение дизайна | 1–2 | Минимальные корректировки архитектуры |
| Реализация задач 1–9 | 9 | Каждая задача выполнялась автоматически по команде «дальше» |
| Контрольные точки | 2 | Проверка работоспособности API и тестов |
| Формирование отчёта | 1 | Генерация итогового отчёта |

**Итого:** ~15–18 итераций от начала до готового проекта.

### Возникшие сложности

1. **Парсинг справочных материалов.** Исходное задание было предоставлено в формате RTF, что затрудняло программный анализ содержимого. Потребовалась ручная конвертация в текстовый формат.

2. **Уточнение требований.** Первоначальная генерация требований не включала volume mount для Docker в режиме разработки — потребовался дополнительный промпт для уточнения.

3. **Балансировка детализации.** Spec-driven подход требует начальных затрат на формализацию, но окупается на этапе реализации — задачи выполняются точнее и с меньшим количеством ошибок.

## Шаг 6. Выводы и рекомендации

### Основные выводы

1. **IDE-агенты предпочтительнее для профессиональной разработки.** Сравнительный анализ показал значительное преимущество IDE-агентов (23–28 баллов) над Cloud-агентами (15–17 баллов). Ключевые факторы — безопасность данных и доступ к контексту проекта.

2. **Безопасность данных — критический фактор.** Cloud-агенты требуют загрузки всего кода в облако, что неприемлемо для проектов с требованиями к конфиденциальности. IDE-агенты работают локально, обеспечивая контроль над данными.

3. **Spec-driven подход структурирует сложные задачи.** Методология Kiro (requirements → design → tasks) позволяет:
   - Формализовать требования до начала кодирования
   - Декомпозировать задачу на управляемые подзадачи
   - Обеспечить трассируемость от требований к коду
   - Автоматизировать последовательное выполнение задач

4. **Начальные затраты на формализацию окупаются.** Spec-driven workflow требует 3–4 итерации на этапе требований, но на этапе реализации задачи выполняются автоматически с минимальным вмешательством.

5. **Качество генерируемого кода зависит от качества спецификации.** Чем точнее сформулированы требования и архитектурные решения, тем качественнее результат. Kiro помогает структурировать этот процесс.

### Рекомендации

#### Когда использовать IDE-агенты (Kiro, Cursor, GitHub Copilot)

- **Любой проект с конфиденциальным кодом** — код остаётся локальным
- **Проекты со сложной архитектурой** — полный доступ к контексту проекта
- **Долгосрочная разработка** — интеграция в существующий workflow
- **Командная работа** — совместимость с Git, CI/CD, code review

#### Когда допустимы Cloud-агенты (Bolt.new, Lovable, Replit Agent)

- **Быстрое прототипирование** — создание MVP за минуты
- **Учебные проекты без требований к безопасности** — быстрый старт без настройки окружения
- **Фронтенд-эксперименты** — визуальный предпросмотр в реальном времени
- **Одноразовые скрипты** — когда код не содержит чувствительных данных

#### Рекомендация по выбору инструмента

| Сценарий | Рекомендуемый инструмент |
|----------|------------------------|
| Production-разработка с требованиями к безопасности | Kiro |
| Быстрая разработка с фокусом на скорость | Cursor |
| Дополнение к существующей IDE (автокомплит) | GitHub Copilot |
| Open-source агент с гибким выбором LLM | Cline |
| Быстрый прототип без требований к безопасности | Bolt.new / Lovable |
| Учебный проект с встроенным хостингом | Replit Agent |

### Заключение

Проведённый сравнительный анализ и практическое применение подтвердили, что для профессиональной разработки с требованиями к безопасности данных и сложной архитектуре оптимальным выбором являются IDE-агенты. Kiro выделяется среди них уникальным spec-driven подходом, который обеспечивает структурированную, воспроизводимую и качественную разработку.
