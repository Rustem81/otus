# Техническое задание: MEXC P2P Агрегатор

## 1. Описание проекта

MEXC P2P Агрегатор — веб-приложение для мониторинга, анализа и ранжирования P2P-объявлений криптобиржи MEXC. Работает в режиме read-only: не совершает торговых операций и не требует API-ключей MEXC.

Приложение решает проблему информационной перегрузки P2P-трейдеров: агрегирует объявления, рассчитывает риск-скор мерчантов, учитывает комиссии при расчёте доходности и предоставляет AI-объяснения для принятия решений.

## 2. Целевая аудитория

| Роль | Описание | Ключевые потребности |
|------|----------|---------------------|
| Начинающий трейдер | Новичок на P2P, 0–6 месяцев опыта | Понятный UI, оценка рисков, подсказки |
| Активный трейдер | Опытный пользователь, 50+ сделок/мес | Скорость, точный расчёт доходности, фильтры |
| Администратор | DevOps / техподдержка | Мониторинг интеграций, управление настройками |

## 3. Функциональные требования

### 3.1. Модуль аутентификации и онбординга

#### FR-1: Регистрация и авторизация
- Регистрация по e-mail с подтверждением
- Авторизация по e-mail + пароль
- Автоматическое истечение сессии после X часов бездействия (конфигурируемо)
- Кнопка «Выход» для обнуления всех активных сессий
- Пароль: минимум 8 символов, буквы + цифры

#### FR-2: Онбординг-мастер
- 4 шага: страна → банки → диапазон сумм → риск-профиль
- Возможность пропустить онбординг (применяются настройки по умолчанию)
- Отображается только при первом входе

#### FR-3: Маркер режима чтения
- В шапке приложения постоянно отображается метка «Read-only: сервис не торгует от вашего имени»

### 3.2. Модуль профиля трейдера

#### FR-4: Банки и способы оплаты
- Выбор из предустановленного списка банков РФ (Сбербанк, Тинькофф, Райффайзен, Альфа-Банк, ВТБ, СБП и др.)
- Возможность добавить кастомный способ оплаты
- Фильтрация объявлений на основе выбранных банков

#### FR-5: Диапазоны сумм и валютные пары
- Задание минимальной и максимальной суммы сделки (RUB)
- Выбор целевых P2P-направлений (MVP: RUB ⇄ USDT)
- Фильтрация объявлений по заданному диапазону

#### FR-6: Риск-профиль
- Три уровня: низкий / средний / высокий
- Влияет на пороги риск-скора при фильтрации и сортировке
- Изменяемый в любой момент в настройках

#### FR-7: Комиссии банка
- Указание комиссии: фиксированная (RUB) и/или процентная (%)
- Используется при расчёте чистой доходности
- По умолчанию: 0 (без комиссии)

#### FR-8: KYC-уровень
- Указание KYC-уровня и страны
- Предупреждение при приближении к типичным лимитам P2P/вывода

### 3.3. Модуль сбора P2P-данных

#### FR-9: Регулярное обновление
- Опрос источника P2P-объявлений с интервалом ≤15 секунд (конфигурируемо)
- Централизованный rate limiting на уровне бэкенда
- Server-side polling (не на клиенте)

#### FR-10: Актуальность объявлений
- Пометка объявлений как «неактивные» в течение 30 секунд после исчезновения из источника
- Проверка актуальности перед отображением в UI

#### FR-11: Нормализация данных
- Сохранение для каждого объявления: ID, цена, объём, лимиты, способы оплаты, валюта, направление (BUY/SELL), метрики мерчанта
- Метрики мерчанта: рейтинг, число сделок, процент успешных, скорость закрытия (при наличии)

#### FR-12: Логирование ошибок интеграции
- Подсчёт ошибок и таймаутов за каждый час
- Доступность в логах и админ-панели

### 3.4. Модуль основного экрана (таблица объявлений)

#### FR-13: Таблица объявлений
- Колонки: мерчант, рейтинг (звёзды), число сделок, цена, спред (%), банки (иконки), лимиты, объём, риск-скор (цветной бейдж), чистая доходность (%)
- Автообновление данных каждые 15 секунд без перезагрузки страницы
- Пустое состояние: «Нет объявлений по заданным критериям» с предложением изменить фильтры

#### FR-14: Фильтрация
- По банкам/способам оплаты (множественный выбор)
- По минимальному рейтингу (слайдер)
- По минимальному числу сделок (слайдер)
- По диапазону суммы (два поля: от/до)
- Фильтры применяются мгновенно (без кнопки «Применить»)

#### FR-15: Сортировка
- По чистой доходности, рейтингу, объёму, риск-скору
- Клик на заголовок колонки: по убыванию → по возрастанию → без сортировки
- Дефолтная сортировка: по интегральному скору (цена + риск + профиль)

#### FR-16: Сохранение фильтров
- Кнопка «Сохранить как по умолчанию»
- Автоприменение сохранённого набора при входе

### 3.5. Модуль риск-скоринга

#### FR-17: Расчёт риск-скора
- Числовой скор от 1 до 10 на основе метрик мерчанта
- Факторы: рейтинг, число сделок, процент успешных, скорость закрытия
- Веса факторов из конфигурации (не из UI)
- Маппинг: 1–3 = низкий (зелёный), 4–7 = средний (жёлтый), 8–10 = высокий (красный)

#### FR-18: LLM-объяснения
- По клику на иконку рядом с риск-скором — всплывающее окно с текстовым объяснением (до 300 символов)
- Модель: GPT-4o-mini
- Кэширование объяснений в Redis (TTL = 10 минут)

#### FR-19: Fallback без LLM
- При недоступности LLM: числовой скор отображается, вместо объяснения — «AI-объяснение временно недоступно»
- Основной flow не блокируется

### 3.6. Модуль чистой доходности

#### FR-20: Расчёт
- Формула: чистая доходность = (спред - комиссия_банка) / цена × 100%
- Комиссия из профиля пользователя (фикс + процент)
- При отсутствии комиссии — расчёт без неё с пометкой

#### FR-21: Отображение
- Колонка «Чистая доходность» в таблице (%)
- Tooltip при наведении: цена, спред, комиссия, итоговый PnL

### 3.7. Модуль карточки объявления

#### FR-22: Детальная карточка
- Модальное окно по клику на строку таблицы
- Полные данные: все метрики мерчанта, банки, лимиты, объём, риск-скор с объяснением
- LLM-резюме «подходит/не подходит» с учётом профиля пользователя

#### FR-23: Переход в MEXC
- Кнопка «Открыть в MEXC» — открывает страницу объявления на MEXC в новой вкладке

### 3.8. Модуль истории и предпочтений

#### FR-24: История просмотров
- Список последних 50 просмотренных объявлений с датой и основными параметрами

#### FR-25: Чёрный список мерчантов
- Кнопка «Заблокировать мерчанта» в карточке объявления
- Объявления заблокированных мерчантов скрываются из таблицы
- Управление чёрным списком в настройках профиля

### 3.9. Модуль администрирования

#### FR-26: Управление валютами и источниками
- Включение/выключение валютных пар
- Включение/выключение источников данных

#### FR-27: Мониторинг
- Дашборд: статус источников (OK / Degraded / Down)
- Счётчики ошибок и таймаутов за 24 часа

#### FR-28: Ролевая модель
- USER — доступ к пользовательскому интерфейсу
- ADMIN — доступ к админ-панели и операционным настройкам

## 4. Нефункциональные требования

### NFR-1: Производительность
- Время отклика бэкенда для списка объявлений: ≤1 секунда (до 500 объявлений)
- Время рендера таблицы в браузере: ≤2 секунды на десктопе
- Интервал обновления данных: ≤15 секунд

### NFR-2: Безопасность
- Хеширование паролей: bcrypt
- Аутентификация: JWT-токены
- Rate limiting на API-эндпоинтах
- CORS: только разрешённые домены
- Валидация входных данных: Pydantic (бэкенд), Zod (фронтенд)
- CSRF-защита

### NFR-3: Надёжность
- Graceful degradation при недоступности LLM
- Graceful degradation при недоступности источника P2P-данных
- Бэкапы БД: ежедневные, хранение ≥7 дней
- Логирование всех 5xx ошибок с уровнем ERROR

### NFR-4: Совместимость
- Браузеры: последние 2 версии Chrome, Firefox, Edge
- Минимальное разрешение: 1024px (адаптивная вёрстка)
- Мобильный режим: упрощённый (может быть отложен)

### NFR-5: Масштабируемость
- Контейнеризация: Docker + Docker Compose
- Stateless бэкенд (сессии в Redis)
- Конфигурация через переменные окружения

## 5. Описание обработки ошибок

### 5.1. Ошибки валидации

| Сценарий | UX-решение |
|----------|-----------|
| Невалидный e-mail при регистрации | Inline-ошибка под полем: «Введите корректный e-mail» |
| Пароль не соответствует требованиям | Inline-ошибка с перечислением требований |
| Сумма вне допустимого диапазона | Слайдер не позволяет выйти за границы; при ручном вводе — ошибка |
| Невалидный формат комиссии | Inline-ошибка: «Введите число от 0 до 100» |

### 5.2. Отсутствие данных

| Сценарий | UX-решение |
|----------|-----------|
| Нет объявлений по фильтрам | Пустое состояние: иконка + «Нет объявлений по заданным критериям» + кнопка «Сбросить фильтры» |
| Нет данных о мерчанте | Прочерк в соответствующих колонках, tooltip «Данные недоступны» |
| Пустая история просмотров | «Вы ещё не просматривали объявления» |
| Нет сохранённых фильтров | Применяются фильтры по умолчанию, подсказка «Настройте и сохраните фильтры» |

### 5.3. Сетевые ошибки

| Сценарий | UX-решение |
|----------|-----------|
| Потеря соединения с бэкендом | Баннер вверху страницы: «Нет соединения с сервером. Данные могут быть устаревшими» + автоматический retry каждые 5 секунд |
| Таймаут запроса к API | Показать последние кэшированные данные + метка «Обновлено X минут назад» |
| Ошибка при сохранении профиля | Toast-уведомление: «Не удалось сохранить. Попробуйте ещё раз» + кнопка retry |
| Источник P2P-данных недоступен | Таблица показывает последние данные + баннер «Данные обновлены X минут назад. Источник временно недоступен» |

### 5.4. Ошибки авторизации

| Сценарий | UX-решение |
|----------|-----------|
| Неверный пароль при входе | «Неверный e-mail или пароль» (без уточнения что именно неверно) |
| Истекшая сессия | Редирект на страницу входа + сообщение «Сессия истекла, войдите снова» |
| Доступ к админке без роли ADMIN | HTTP 403 + страница «Доступ запрещён» |
| Попытка изменить чужой профиль | HTTP 403 + «Недостаточно прав» |

### 5.5. Ошибки LLM-интеграции

| Сценарий | UX-решение |
|----------|-----------|
| LLM-сервис недоступен | Числовой риск-скор отображается, вместо объяснения — «AI-объяснение временно недоступно» |
| Таймаут LLM-запроса (>5 сек) | Прервать запрос, показать скор без объяснения |
| Превышение лимита API OpenAI | Логирование ошибки, fallback на скор без объяснения, уведомление администратору |
| Некорректный ответ LLM | Логирование, fallback на скор без объяснения |
