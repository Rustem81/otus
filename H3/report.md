## Шаг 1. Формулировка продуктовой идеи

Подробное описание: **[project_description.md](project_description.md)**

**Идея:** MEXC P2P Агрегатор — веб-приложение для мониторинга и анализа P2P-объявлений криптобиржи MEXC в режиме read-only.

**Проблема:** P2P-трейдеры тратят время на ручное сравнение объявлений, не имеют интегральной оценки рисков и не учитывают скрытые комиссии при оценке доходности.

**Целевая аудитория:** начинающие и активные P2P-трейдеры (РФ, RUB ⇄ USDT), администраторы сервиса.

**Ключевые функции MVP:**
1. Агрегация и фильтрация P2P-объявлений (автообновление ≤15 сек)
2. Риск-скоринг мерчантов с LLM-объяснениями (GPT-4o-mini)
3. Расчёт чистой доходности с учётом комиссий банка

## Шаг 2. Генерация UI-концепций

Подробное описание с промптами: **[ui_concepts/README.md](ui_concepts/README.md)**

Для генерации использованы Text-to-UI инструменты (v0.dev).

### Сгенерированные концепции

| # | Стиль | Описание | Плюсы | Минусы |
|---|-------|----------|-------|--------|
| 1 | Минималистичный (Light) | Таблица + боковые фильтры, белый фон | Максимум данных на экране, быстрое сканирование | Может выглядеть «скучно» |
| 2 | Карточный (Dark) | Карточки в сетке, тёмная тема | Современный крипто-вид, визуально выделенные объявления | Меньше объявлений на экране |
| 3 | Корпоративный с графиками | График спреда + таблица + профиль | Максимум аналитики, профессиональный вид | Перегруженность для MVP |

### Выбор: Концепция 1 — Минималистичный дашборд

**Обоснование:** Табличный формат обеспечивает максимальную плотность информации для сравнения 10–20 объявлений. Идеально ложится на Quasar QTable. Проще реализовать в MVP. Элементы из других концепций (тёмная тема, графики) можно добавить позже.

## Шаг 4. Разработка критериев приёмки

Критерии приёмки написаны для каждой User Story в формате Given/When/Then. Полный список: **[user_stories.md](user_stories.md)**.

### Пример (US-5: Риск-скор и LLM-объяснение)

```gherkin
Given объявление в таблице
When пользователь смотрит колонку «Риск»
Then отображается числовой скор (1–10) с цветовой индикацией (зелёный/жёлтый/красный)

Given пользователь кликает на иконку рядом с риск-скором
When LLM доступен
Then появляется всплывающее окно с текстовым объяснением (до 300 символов)

Given пользователь кликает на иконку рядом с риск-скором
When LLM недоступен
Then отображается «AI-объяснение временно недоступно» и числовой скор остаётся видимым
```

### Покрытие

| Функция | Позитивные сценарии | Негативные сценарии |
|---------|--------------------|--------------------|
| Регистрация (US-1) | Успешная регистрация | Дубликат e-mail, слабый пароль |
| Онбординг (US-2) | Прохождение всех шагов | Пропуск онбординга |
| Таблица (US-3) | Отображение данных, автообновление | Пустой список |
| Фильтрация (US-4) | Фильтр по банку, рейтингу, сумме | — |
| Риск-скор (US-5) | Отображение скора, LLM-объяснение | LLM недоступен |
| Доходность (US-6) | Расчёт с комиссией, tooltip | Комиссия не указана |
| Сохранение фильтров (US-7) | Сохранение, автоприменение | — |

## Шаг 5. Проектирование обработки ошибок

Подробное описание: раздел 5 в **[technical_specification.md](technical_specification.md)**.

### Категории ошибок

#### 1. Ошибки валидации
- Невалидный e-mail → inline-ошибка под полем
- Слабый пароль → перечисление требований
- Сумма вне диапазона → слайдер ограничивает + ошибка при ручном вводе

#### 2. Отсутствие данных
- Нет объявлений по фильтрам → пустое состояние с иконкой + «Сбросить фильтры»
- Нет данных о мерчанте → прочерк + tooltip «Данные недоступны»

#### 3. Сетевые ошибки
- Потеря соединения → баннер «Нет соединения» + retry каждые 5 сек
- Таймаут API → показать кэшированные данные + метка «Обновлено X мин назад»
- Источник P2P недоступен → последние данные + баннер о недоступности

#### 4. Ошибки авторизации
- Неверный пароль → «Неверный e-mail или пароль» (без уточнения)
- Истекшая сессия → редирект на логин + сообщение
- Доступ без прав → HTTP 403 + «Доступ запрещён»

#### 5. Ошибки LLM-интеграции
- LLM недоступен → скор без объяснения (graceful degradation)
- Таймаут >5 сек → прервать, показать скор
- Превышение лимита API → логирование + fallback + уведомление админу

## Выводы

1. **Преемственность проектов.** H3 является логическим продолжением H2 (MEXC P2P Агрегатор). В H2 были созданы steering-правила и промпт-шаблоны, в H3 — UI-концепции, User Stories и ТЗ для того же проекта.

2. **Text-to-UI инструменты** позволяют быстро визуализировать идеи и сравнить подходы к дизайну до начала разработки. Концепции различаются именно визуальным стилем (как рекомендовано на лекции), а не функциональностью.

3. **User Journey в табличном формате** (этапы → цели → действия → точки касания → эмоции → болевые точки → улучшения) даёт полную картину взаимодействия пользователя с продуктом. Негативные сценарии проработаны отдельно от позитивных.

4. **User Story Mapping** структурирует функциональность вокруг пользовательских ролей и сценариев. 14 User Stories покрывают MVP и ближайшие итерации, приоритизированы по Must/Should/Nice to Have.

5. **Given/When/Then** формат критериев приёмки делает требования проверяемыми и однозначными — готовыми для передачи AI-инженеру или QA.

6. **Проектирование обработки ошибок** на этапе ТЗ (а не при разработке) предотвращает «слепые зоны» в UX. Для каждой ключевой функции описаны негативные сценарии и UX-решения.

7. **ТЗ готово для работы с AI-инженером** — структурировано по модулям, содержит 28 функциональных и 5 нефункциональных требований, описание ошибок и критерии приёмки.
