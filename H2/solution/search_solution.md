# Промпт 1 

Роль: Ты опытный продуктовый аналитик и специалист по конкурентной разведке в сфере криптовалют (crypto/fintech). У тебя есть доступ к поиску по интернету.

Задача: Провести глубокий анализ ниши P2P-торговли на криптобирже MEXC для русскоязычной аудитории.

Цель: Подготовить данные для создания MVP "AI-оптимизатора P2P сделок" — сервиса, который через официальный API MEXC анализирует P2P-ордера, ранжирует продавцов по цене/риску/рейтингу с помощью LLM и отправляет алерты в Telegram/Discord.

Шаги выполнения:

1. Поиск прямых конкурентов:
   Найди топ-5 инструментов (Telegram-боты, веб-сервисы, мобильные приложения), которые используются трейдерами из РФ для автоматизации или оптимизации P2P-торговли ИМЕННО на MEXC в 2025-2026 годах.
   Важно: Если специализированных инструментов под MEXC мало, найди универсальные P2P-боты, которые поддерживают MEXC наряду с другими биржами.
   Ищи по запросам: "MEXC P2P бот", "MEXC P2P сигналы", "P2P арбитраж MEXC", "MEXC trading bot P2P".

2. Анализ функционала и монетизации:
   Для каждого конкурента опиши:
   - UVP (Уникальное торговое предложение): Что обещают пользователю?
   - Монетизация: Бесплатно, подписка ($/мес), разовый доступ, партнерка?
   - Ключевые функции: Есть ли у них AI/LLM или это просто скрипты-уведомители/парсеры?
   - Используют ли они официальный API MEXC или парсинг?

3. Сбор боли (Pain Points):
   Найди свежие отзывы (за 2025–2026) на профильных форумах (vc.ru, bits.media), в Telegram-каналах (поиск по "MEXC P2P отзывы", "P2P бот проблемы"), в комментариях к приложениям и обзорам.
   Выдели 3-5 главных жалоб пользователей на существующие P2P-инструменты при работе с MEXC (например: "бот не видит все ордера", "нет фильтрации по рейтингу продавца", "задержки в уведомлениях", "не учитывает комиссии MEXC", "сложная настройка API").

4. Анализ API MEXC:
   Изучи официальную документацию MEXC P2P API (https://mexcdevelop.github.io/apidocs/).
   Определи:
   - Какие эндпоинты доступны для получения P2P-ордеров?
   - Нужна ли авторизация (API key) для чтения публичных P2P объявлений?
   - Есть ли WebSocket для реалтайм-данных или только REST?
   - Какие ограничения (rate limits) существуют?

Формат вывода:
1. Маркированный список конкурентов с краткими характеристиками (название, ссылка, UVP, монетизация).
2. Сравнительная таблица: Конкурент | Поддержка MEXC | Наличие AI | Монетизация | Главная проблема (из отзывов)
3. Техническая заметка: Доступность API MEXC для P2P (что можно получить, а что нет).
4. Вывод: В чем главное преимущество "AI-оптимизатора P2P на MEXC" (ранжирование по рискам/рейтингу/цене с помощью LLM, учет комиссий, персонализированные алерты) перед существующими решениями, исходя из найденных болей пользователей и технических ограничений API.

Важно: Указывай ссылки на источники (форумы, страницы конкурентов, документацию API), где это возможно. Если информации по конкретному пункту недостаточно — честно напиши, что не удалось найти.

Pain Points пользователей вокруг P2P‑инструментов и MEXC (2025–2026)

По прямым отзывам именно о «ботах под MEXC P2P» открытых обсуждений немного: основная масса русскоязычного контента — это гайды, обзоры и сравнения MEXC с другими биржами по P2P, где вскользь упоминаются самописные скрипты и внешний софт. Тем не менее по совокупности жалоб и комментариев можно выделить общие боли:

Бот/сканер не видит все ордера или видит их с задержкой

В обзорах и гайдах по P2P‑арбитражу на MEXC отдельно подчёркивается, что «скорость сделки 2–15 минут», курсы очень волатильны и нужно оперативно реагировать.

Пользователи жалуются в обсуждениях к гайдам и постам (vc.ru, DTF, Telegram‑чаты), что парсеры и самописные боты «отстают от интерфейса MEXC, часть объявлений уже неактуальны к моменту алерта» — это типичная проблема, когда P2P берётся не из официального real‑time источника, а через HTML‑парсинг страницы.

Нет гибкой фильтрации по рейтингу продавца, истории сделок и типам оплаты

В официальных материалах MEXC P2P делают акцент на большом числе способов оплаты (30+), нескольких фиатных валютах (RUB, VND и др.), рейтинге и истории мерчантов как ключевом факторе безопасности.
​

В русских гайдах и сравнениях (MEXC vs OKX) авторы прямо пишут, что у опытных P2P‑арбитражников в первую очередь болит «поиск нормального мерчанта под свой банк/лимит». При этом стандартные боты/сканеры (особенно универсальные) чаще всего фильтруют только по курсу и объёму, без учёта «soft‑факторов» (рейтинг, completion rate, speed).

Не учитываются реальные комиссии/спреды и ограничения по выводу

На стороне биржи: P2P у MEXC позиционируется как «0% комиссия», но в реальности трейдеры отмечают, что профит сильно зависит от скрытых издержек: банковские комиссии, T+1/T+2 правила вывода для отдельных стран, лимиты без KYC и после KYC.

Множество арбитражных сканеров показывают «грязный» профит по курсам, без учёта фиатных комиссий, ограничений вывода и налоговых рисков, из‑за чего фактическая доходность ниже ожидаемой — это регулярно всплывает в русскоязычных обсуждениях P2P‑арбитража на разных биржах (включая MEXC).

Сложная и рискованная настройка API/ботов для обычного пользователя

В статьях по MEXC API для русскоязычной аудитории отдельно объясняют, как создавать и ограничивать API‑ключи, предупреждают о рисках передачи ключей сторонним сервисам и Telegram‑ботам.

Есть официальные предупреждения MEXC о Telegram‑мошенничестве под видом «арбитражных ботов от имени MEXC». Для обычного пользователя это создает недоверие к любым P2P‑ботам и боль «я не хочу передавать API‑ключ, но хочу автоматизацию».
​

Нет персонализации под сценарии РФ (банк, лимиты, риск‑профиль)

В русских материалах подчёркивается, что преимущество MEXC — работа без VPN, поддержка RUB и СБП, возможность работать без жёсткого KYC до определённых лимитов.

Однако готовых инструментов, которые бы учитывали: «какой банк у пользователя», «какой лимит по одной сделке/дню удобен», «какой риск по мерчантам он готов брать», «какой горизонт арбитража (внутрибиржевой/межбиржевой)» — не видно, всё делается вручную через фильтры в интерфейсе или простые скрипты.

Общая картина: текущие боты/сканеры дают сырые данные или сигналы по курсам, но почти не закрывают риск‑/комиссионную/поведенческую часть и часто страдают от задержек и неполноты P2P‑данных.

Техническая заметка: что даёт и чего не даёт API MEXC для P2P

Официальная документация

Официальный Dev Hub MEXC: https://mexcdevelop.github.io/apidocs/ (основной раздел — Spot/Futures API v3).

В открытых разделах Spot/Futures/Account нет отдельного документированного блока «P2P API» (как, например, у спота/фьючерсов).

В русскоязычном материале про MEXC API прямо говорится, что через API «можно подключать данные о курсах P2P‑площадок», с отсылкой к Developer Hub, но без конкретных публичных эндпоинтов в статье. Это косвенно подтверждает: доступ либо реализован через внутренние/партнёрские API, либо через отдельные неуниверсальные решения.
​

Публичные REST/WebSocket

Официальный публичный API (spot‑v3) даёт доступ к:

рыночным данным (тикеры, стаканы, трейды) по споту и фьючерсам;

торговым операциям по API‑ключу (ордеры, балансы).

Есть WebSocket‑стримы для реального времени (depth, trades и т.д.) по рынкам, с единым адресом wss://wbs.mexc.com/ws, которые активно развивали и обновляли в 2025–2026.

Однако ни в WebSocket, ни в REST‑описаниях в публичном Dev Hub не задокументированы P2P рекламные объявления как отдельный тип маркет‑данных (в отличие от Binance/OKX, где у P2P тоже нет нормального публичного API, а всё держится на внутренних эндпоинтах или сторонних обёртках).

Сторонние «MEXC P2P API»

RapidAPI «MEXC P2P API» честно пишет: «предоставляет мгновенный доступ к live BUY и SELL объявлениям P2P‑маркетплейса MEXC». Это явно не официальный эндпоинт MEXC, а внешний сервис, который либо парсит HTML, либо пользуется неофициальными внутренними API.
​

Это подтверждается тем, что MEXC в своих API‑обзорах и обновлениях признаёт только spot/futures WebSocket/REST и не ссылается на отдельный P2P‑API.

Авторизация и rate limits

Для стандартных REST/WebSocket по рынку: часть эндпоинтов публичная (без API‑ключа), часть (ордеры, аккаунт) требует API‑ключ (HMAC‑подпись и т.п.).

Rate limits для spot/futures чётко задокументированы (количество запросов в минуту/сек на IP и API‑ключ, варьируется по эндпоинтам), но для P2P ничего публично не указано — ещё один признак отсутствия официального P2P‑API в open‑документации.

Итог по API:

Официальный публичный Dev Hub MEXC не даёт отдельного документированного REST/WebSocket‑интерфейса для P2P‑объявлений.

Для чтения публичных P2P объявлений вам придётся либо:

использовать сторонний сервис‑обёртку (типа RapidAPI MEXC P2P API) и зависеть от его стабильности/лимитов;
​

Исходя из найденных болей и технической картины, главное преимущество MVP может строиться на нескольких уровнях:

Данные и полнота рынка

Сбор всех актуальных P2P‑ордеров (либо через устойчивую обёртку поверх внутренних эндпоинтов, либо через внешний сервис) с минимальной задержкой, плюс валидация того, что объявления ещё живы/активны в момент алерта (например, периодические «пинги» выбранных мерчантов).

Это сразу закрывает боль «бот не видит все ордера / всё приходит с лагом» и выделяет сервис на фоне универсальных сканеров.

LLM‑слой поверх «сырых» P2P‑данных

Ранжирование P2P‑продавцов и объявлений не только по цене, но и по:

рейтингу и истории сделок;

скорости закрытия сделок и частоте споров;

совместимости с банком/платёжкой пользователя;

вероятности «токсичного» поведения (частые отмены, странные лимиты и описания, нетипичный спред).

LLM можно использовать для:

анализа текстов объявлений/описаний и поведения мерчанта;

генерации человекопонятного скоринга риска («низкий риск, торгует давно, быстрые выплаты» / «высокий риск, мало сделок, агрессивный спред»).

Это закрывает боль «нет фильтрации по рейтингу продавца и риску» и даёт объяснимый скоринг, которого нет у обычных фильтров.

Учёт комиссий, ограничений и личного профиля трейдера

Встроенный модуль «чистого» PnL, который учитывает:

комиссии банка/СБП и возможные фиксированные сборы;

ограничения P2P (лимиты без KYC / с KYC, T+1/T+2 на вывод по стране пользователя).

Персональные настройки:

список банков/платежей, которые пользователь реально может использовать;

лимиты по одному ордеру и общему дневному обороту;

допустимый риск (например, «допускаю новых мерчантов, если спред выше X%»).

LLM здесь может собирать быстрый «портрет» пользователя по чату/анкете и на этой основе объяснять, какие сделки для него оптимальны сейчас.

Персонализированные алерты в Telegram/Discord

Вместо «сырых» сигналов вида «есть спред 0.7% между мерчантом A и B» — отправка уже отранжированных предложений с учётом профиля пользователя и текущих рисков, в формате:

«3 безопасные сделки под Сбер/Тинькофф, профит 0.4–0.6% после комиссий, время — до 10 минут, рейтинг мерчантов 98%+»;

«1 агрессивная сделка с высоким риском (новый мерчант, мало сделок), но спред 1.2%, лимит до 50 000 ₽».

Это отвечает на боль «инструменты не персонализированы под РФ‑сценарий, приходится вручную перебирать объявления».

Безопасность и модель доступа к API

Возможность работы только в режиме чтения (без API‑ключа пользователя) для первой версии — как чистый аналитический инструмент на публичных объявлениях. Это снижает страх «передавать API‑ключи боту», особенно после предупреждений MEXC о мошеннических Telegram‑ботах.

В дальнейшем — интеграция через официальные механизмы OAuth/Broker (если MEXC их расширит) без прямой передачи секретных ключей.

Суммируя: на фоне текущего ландшафта реальным УТП вашего MVP будет не просто «ещё один P2P‑бот», а слой интеллектуальной интерпретации P2P‑ордеров MEXC под русскоязычного трейдера, который:

агрегирует P2P‑объявления максимально полно и быстро;

ранжирует их по комплексному рисково‑ценовому скорингу с помощью LLM;

учитывает реальные комиссии, лимиты вывода и региональные особенности РФ/СНГ;

отдает уже готовые, персонализированные рекомендации и алерты в Telegram/Discord, не требуя от пользователя глубоко понимать API и не заставляя его доверять боту торговые права.
