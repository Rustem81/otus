# Аудит ТЗ: MEXC P2P Агрегатор (MVP)

---

## 1. Противоречия и конфликты

### 1.1. Опрос каждые 15 сек vs лаг ≤ 30 сек (п. 3.1 + п. 0)

Если поллинг идёт раз в 15 секунд, а объявление может «жить» ещё 30 секунд после исчезновения из источника (п. 3.2), то реальный лаг для пользователя может достигать 15 + 30 = **45 секунд**, что нарушает заявленное ограничение «не более 30 секунд» из п. 0.

**Рекомендация:** Либо сократить TTL «неактивных» объявлений до 15 сек, либо увеличить частоту поллинга до 10 сек, либо честно указать целевой лаг ≤ 45 сек.

### 1.2. Telegram-OAuth в MVP vs «без API-ключей» (п. 1.1 + п. 0)

ТЗ заявляет «MVP не требует API-ключей MEXC», но Telegram-OAuth — это отдельная интеграция, требующая регистрации бота, хранения токенов и обработки callback. Для MVP это дополнительная поверхность атаки и сложность.

**Рекомендация:** Оставить Telegram-OAuth как опциональную фичу v1.1. Для MVP достаточно e-mail + пароль.

### 1.3. LLM-резюме в карточке (п. 7.3) vs кэш LLM-объяснений (п. 5.2)

В п. 5.2 кэш LLM-объяснений привязан к «мерчанту/объявлению» на 10 минут. Но в п. 7.3 LLM-резюме должно учитывать **профиль конкретного пользователя** («подходит/не подходит вам»). Это значит, что кэш по мерчанту не подходит — нужен кэш по паре (мерчант + профиль пользователя), что кратно увеличивает объём кэша и количество вызовов LLM.

**Рекомендация:** Для MVP оставить только один тип LLM-вывода — объяснение риск-скора (п. 5.2), без персонализированного резюме. Персонализированное «подходит/не подходит» вынести в v1.1.

### 1.4. «Множественные пресеты профиля — в следующую версию» (п. 2.3) vs сохранение фильтров (п. 4.4)

П. 2.3 явно откладывает множественные пресеты, но п. 4.4 позволяет сохранить «сценарий по умолчанию». Это непротиворечиво, но граница размыта: если пользователь хочет 2 сценария фильтров — это уже «пресет» или нет? Нужно чётко определить: в MVP — **один** сохранённый набор фильтров, без переключения.

### 1.5. Спред в таблице (п. 4.1) — относительно чего?

Колонка «спред» упоминается, но нигде не определено: спред к чему? К биржевому курсу USDT/RUB на споте MEXC? К среднему по P2P? К курсу ЦБ? Без определения базовой цены «спред» — бессмысленная метрика.

**Рекомендация:** Явно определить reference price (например, средневзвешенная цена по топ-10 объявлениям или спотовый курс MEXC) и формулу расчёта спреда.

## 2. Пропуски (что не хватает для запуска MVP)

### 2.1. Безопасность

| Пропуск | Критичность | Комментарий |
|---------|-------------|-------------|
| Подтверждение e-mail при регистрации | Высокая | Без верификации — спам-аккаунты и злоупотребления |
| Хеширование паролей (bcrypt/argon2) | Высокая | Не упомянуто, но подразумевается — нужно зафиксировать |
| CSRF-защита | Высокая | Веб-приложение с сессиями — обязательно |
| Rate limiting на login/register | Высокая | Защита от brute-force |
| HTTPS only | Высокая | Не упомянуто явно |
| Content Security Policy (CSP) | Средняя | Защита от XSS |
| Политика хранения персональных данных | Высокая | E-mail, банки, KYC-уровень — это ПД, нужна хотя бы базовая privacy policy |

### 2.2. Инфраструктура и мониторинг

| Пропуск | Критичность | Комментарий |
|---------|-------------|-------------|
| Health check эндпоинт | Высокая | Для мониторинга и балансировщика |
| Метрики приложения (Prometheus/аналог) | Средняя | Без метрик невозможно отслеживать SLA из п. 10 |
| Стратегия деплоя (CI/CD) | Средняя | Не описана |
| Миграции БД | Средняя | Не упомянуты, но критичны для эволюции схемы |

### 2.3. Обработка ошибок на UI

Нигде не описано поведение UI при:
- Недоступности бэкенда
- Таймауте загрузки данных
- Пустом списке объявлений (нет данных по паре)
- Ошибке LLM-сервиса (п. 5.3 описывает fallback для скора, но не для UI-состояния)

**Рекомендация:** Добавить раздел «Состояния ошибок и пустые состояния UI».

### 2.4. Конфигурация и переменные

В ТЗ много «X» без значений:
- «сессия истекает через X часов» (п. 1.1)
- «LLM-объяснение до X символов» (п. 5.2)
- «не более 1 секунды при объёме до X объявлений» (п. 10)
- «не более 2 секунд при X объявлениях» (п. 10)
- «последних N просмотренных» (п. 8.1)

**Рекомендация:** Зафиксировать конкретные значения для MVP (например: сессия = 24ч, LLM = 300 символов, таблица до 200 объявлений, история = 50 записей).

### 2.5. Стек технологий

ТЗ не фиксирует технологический стек. Для MVP это критично, т.к. влияет на оценку сроков и ресурсов.

**Рекомендация:** Добавить раздел с целевым стеком (например: React/Next.js, Node.js/Python, PostgreSQL, Redis).

### Ядро MVP (must-have)

1. Регистрация/логин по e-mail + сессии
2. Профиль: банки, диапазон сумм, одна валютная пара, риск-профиль
3. Сбор P2P-данных с поллингом и нормализацией
4. Таблица объявлений с фильтрами и сортировкой
5. Риск-скор (rule-based) + LLM-объяснение с fallback
6. Расчёт чистой доходности с учётом комиссий
7. Карточка объявления + кнопка «Открыть в MEXC»
8. Ролевая модель USER/ADMIN
9. Маркер «Read-only»

## 4. Переработанное ТЗ (MVP)

---

### 0. Общие ограничения MVP

- MVP работает **только с публичными P2P-объявлениями MEXC** и **не совершает торговых операций** от имени пользователя.
- API-ключи MEXC **не требуются и не используются**.
- Данные P2P-объявлений получаются из стороннего API / неофициальных эндпоинтов с централизованным rate limiting и логированием ошибок на бэкенде.
- Целевой лаг данных на UI: **не более 30 секунд** от фактического состояния на MEXC (настраиваемый параметр; при поллинге раз в 10 сек и TTL неактивных = 15 сек).
- Приложение доступно **только по HTTPS**.
- Все пароли хранятся в виде хешей (bcrypt/argon2id).
- Персональные данные (e-mail, настройки банков) защищены в соответствии с базовой политикой конфиденциальности.
